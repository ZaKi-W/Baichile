# 白吃了 · 腾讯云开发（CloudBase）迁移方案

> 目标：将当前 pnpm monorepo（NestJS API + uni-app 客户端 + Vue 后台）从「本地 Docker Compose」整体迁移到腾讯云开发平台。
> 范围：部署平台与后端托管迁移，**不重写业务代码架构**。
> 状态：方案文档，不含任何代码改动。

---

## 0. 先讲结论（推荐方案）

**采用「云托管 CloudBase Run + 云数据库 PostgreSQL 托管实例 + 云存储」的混合架构，而不是把后端重写为「云函数 + 文档数据库」。**

这本质上是一次 **部署平台迁移（lift-and-shift + 选择性增强）**，不是应用架构重写。它和项目技术方案文档第 3.2 节「不采用 CloudBase 作为主架构」并不矛盾——那一条反对的是「用云函数 + 文档库作为**应用架构**」，而我们这次是把 CloudBase 作为**托管平台**来用，业务仍跑在 NestJS + 关系数据库上。

**为什么不全量重写为云函数 + 云数据库：**

| 维度 | 云函数 + 文档库（不推荐） | 云托管 + 托管 PG（推荐） |
|---|---|---|
| 现有代码 | 全部 NestJS/TypeORM 代码需重写 | 原样容器化部署，改动极小 |
| 关系数据 / 事务 | 文档库弱事务，订单/钱包重写成本高 | PostgreSQL 原生支持 |
| 未来 PostGIS 空间能力 | 文档库不支持，需自己算距离 | 托管 PG 可直接开 PostGIS |
| 跨端统一账号 / 数据采集审核 | 云函数拆分复杂 | NestJS 模块化单体照常 |
| 微信登录 / 支付 | 需云函数 + 密钥 | 云托管内可用「微信云调用」免密钥 |
| 冷启动 / 长连接 | 云函数冷启动明显 | 云托管常驻容器，稳定 |
| 回滚 | 重写不可逆成本高 | 保留本地 Docker 环境，随时切回 |

**一句话**：保留你已有的工程投资，把"服务器"换成 CloudBase。

## 1. 现状评估

### 1.1 架构与代码组织

- **Monorepo（pnpm workspace）**：`apps/client`（uni-app Vue3 + TS + Vite + Pinia，首发 mp-weixin）、`apps/api`（NestJS 11 + Fastify 5 + TypeORM 0.3）、`apps/admin`（Vue Web 后台）、`packages/*`（api-contract、domain、map-core、ui-tokens、icon-registry、catalog-schema、config、map-core）。
- **API 是一个模块化单体**，模块边界清晰：`auth / account / catalog / order / delivery / map / asset / analytics / admin / health`。当前 `src` 实际目录为 `database / admin / scripts / share`，模块以 service 形式组织（如 `order.service.ts`）。
- **客户端跨端**：同一套 Vue 代码编译到 mp-weixin / H5 / App；API 地址由 `apps/client/src/config/api.ts` 的 `VITE_API_BASE_URL` 决定（开发态默认 `http://127.0.0.1:3000`）。

### 1.2 技术栈

| 层级 | 当前实现 |
|---|---|
| 客户端 | uni-app（Vue3 + TS），构建 mp-weixin / h5 / app |
| 后端 | NestJS + Fastify，TypeORM，JWT（access/refresh） |
| 数据库 | PostgreSQL 17（Docker `postgres:17-alpine`），**当前未实际使用 PostGIS**（仅有 `coordSystem` 字符串字段，无 `ST_` 空间查询） |
| 缓存/队列 | 规划中（Redis/Valkey + BullMQ，第二阶段） |
| 文件/图片 | **本地静态资源**：`apps/client/static/choutuan-img/` 下 **760 张 webp**，随小程序包分发 |
| 后台 | 独立 Vue SPA，接口 `/v1/admin` |
| 第三方 | 微信小程序（AppID 已配置：`wxaeaa8efe4671ce80`）、腾讯位置服务（Key 已配置） |

### 1.3 数据存储方式

- **关系数据（TypeORM 实体，PostgreSQL）**：`account / address / admin-audit-log / admin-session / admin-user / analytics-event / category / menu-item / share-config / share-invite / store-sub-category / store / virtual-order / visitor-session / wallet-transaction`，外加 `store_locations`（通过 `store.locationId` 关联）。
- **Schema 由 TypeORM migration 管理**：已有 7 个迁移文件（`1760000000000` ~ `1760000006000`），覆盖持久化表、目录与埋点、卡路里、钱包、配送异常、后台控制台、分享奖励。
- **种子数据**：`db:seed` 可重复执行；商品目录来自数据库，非硬编码。
- **运行时数据**：游客 token、refresh token、订单进度（以 `startedAt + durationMs` 为唯一事实来源）。
- **图片**：当前**打包进小程序**（760 webp 在 static 目录，并复制到 `dist/**/static/choutuan-img`）。这本身已逼近小程序包体约束，是迁移的额外收益点——应外链到云存储/CDN。

### 1.4 第三方服务依赖

| 依赖 | 当前状态 | 迁移影响 |
|---|---|---|
| 微信小程序登录 | AppID/Secret 已在 `.env`，`wx.login` code 换身份 | 可升级为「微信云调用」，免存 Secret |
| 腾讯位置服务 | Key 已配置，服务端 `/v1/map/*` 调用 | 外部 HTTP，云托管内照常调用，无需变动 |
| 微信支付 | 未接入（订单 100% 虚拟） | 未来如需，云托管 + 云调用更顺 |
| COS + CDN | 仅规划，未使用 | 由云存储替代 |
| Redis/BullMQ | 未接入 | 后续用 TencentDB Redis + 云托管 |

## 2. 迁移策略

### 2.1 迁移范围

- **纳入**：API 容器托管、PostgreSQL 数据库、760 张图片资源、Admin 后台、客户端 API 指向、微信登录接入方式。
- **不纳入（保持不变）**：uni-app 客户端业务代码、NestJS 业务逻辑、TypeORM 实体、PostgreSQL 关系模型、腾讯位置服务调用、项目 monorepo 结构。

### 2.2 渐进式 vs 全量

- **不做全量重写**（见第 0 节理由）。
- **采用渐进式平台迁移**：本地 Docker 环境全程保留作为「黄金副本与回滚基线」，CloudBase 上并行搭建，验证通过后切换流量。因为是 MVP 阶段、数据为虚拟订单与开发态数据，**无需双写/灰度数据同步**，采用「维护窗口 + 一次性数据导出导入」即可。

### 2.3 推荐落地路径（5 阶段）

| 阶段 | 内容 | 产出 | 风险等级 |
|---|---|---|---|
| P0 准备 | 开通 CloudBase 环境；创建云托管服务、云数据库 PostgreSQL 实例（开启 PostGIS 扩展备用）、云存储桶；本地 Docker 环境 `pg_dump` 全量快照 | 环境就绪 + 本地快照 | 低 |
| P1 数据 | 在托管 PG 上跑现有 migration；导入种子数据；导入开发态真实数据 | 云端库与本地一致 | 中 |
| P2 后端 | 为 `apps/api` 新增 Dockerfile；配置云托管（环境变量、私有网络连通 PG）；部署；健康检查 | 云托管 API 可用 | 中 |
| P3 资源 | 写脚本将 760 张 webp 上传云存储，回写 `assets.objectKey`/URL；客户端改为 CDN 外链 | 图片外链化 | 中 |
| P4 接入 | 客户端 `VITE_API_BASE_URL` 指向云托管域名并重新构建；小程序后台配置 request 合法域名；Admin 部署到云托管/静态托管；微信登录改云调用 | 全链路跑通 | 中 |

### 2.4 优先级

1. **最高**：P0 环境 + P1 数据库（没有数据一切免谈）。
2. **次高**：P2 云托管 API（核心后端上线）。
3. **高**：P4 客户端指向（决定用户能否访问）。
4. **中**：P3 图片外链（影响包体与加载，但不阻塞功能）。
5. **可选增强**：微信云调用、未来 Redis。

## 3. 云开发服务映射

| 现有组件 | 腾讯云开发对应服务 | 说明 |
|---|---|---|
| NestJS + Fastify（Docker 容器） | **云托管 CloudBase Run** | 容器化部署，支持自定义域名、HTTPS、私有网络；常驻实例，无云函数冷启动痛点 |
| PostgreSQL 17（Docker volume） | **云数据库 PostgreSQL（托管实例）** | 同版本托管，自动备份；按需开启 PostGIS 扩展 |
| Docker 本地 volume 持久化 | 托管 PG 自动备份 + 跨可用区 | 运维下沉给平台 |
| 本地 static 760 webp | **云存储（Cloud Storage）+ CDN** | 对象存储，提供默认域名或自定义域名 + CDN 加速 |
| COS + CDN（规划未用） | **云存储**（同一家族，统一） | 不再单独采购 COS |
| Redis/Valkey + BullMQ（二期） | **TencentDB Redis + 云托管** | 云托管容器内连接托管 Redis |
| 埋点 `analytics_event` | 云数据库 PostgreSQL（同库）或 **云数据库（文档）** | 建议同 PG 保持事务一致；海量写入时可拆到文档库 |
| 微信登录（Secret 换身份） | **微信云调用（云托管内）** | 免存 AppSecret，安全性更高 |
| 腾讯位置服务调用 | 保持不变（云托管内出公网 HTTP） | 外部依赖，不涉及 CloudBase |
| Admin Vue SPA | **云托管（Web 服务）/ 静态网站托管** | 构建产物部署到 CloudBase |
| 本地 `pg_dump` 快照 | 云数据库自动备份 + 手动快照 | 回滚基线 |

> 注：**云函数（SCF）在本方案中不作为主后端**。仅在两个边缘场景可考虑：① 微信订阅消息推送等轻量触发器；② 图片处理/审核等异步任务。其余一律走云托管，避免架构分裂。

## 4. 数据迁移计划

### 4.1 数据库 Schema 迁移

- **方式**：不导出 schema DDL，而是**直接在新托管 PG 上重跑 TypeORM migration**（`pnpm --filter @baichile/api db:migrate`）。原因：migration 是单一事实来源，手工 dump 会带入本地版本差异。
- **步骤**：
  1. 创建托管 PG 实例，建库 `baichile`，开启 `postgis` 扩展（备用）。
  2. 配置 `DATABASE_URL` 指向托管 PG。
  3. 执行 `db:migrate` → 得到与本地完全一致的结构。
  4. 执行 `db:seed` → 导入 12 店/156 菜等种子数据。

### 4.2 开发态真实数据迁移（可选）

- 当前本地仅有开发测试数据（游客 token、少量虚拟订单）。若需保留：
  1. 本地 `pg_dump --data-only --exclude-table=*_migrations baichile > dump.sql`。
  2. 托管 PG `psql` 导入。
  3. **注意**：`refresh_tokens` / `visitor_sessions` 属运行时态，建议清空重发，避免密钥残留。

### 4.3 图片资源迁移（760 webp）

- **问题**：图片当前在 `apps/client/static/choutuan-img`，打包进小程序。需外链。
- **步骤**：
  1. 用 CloudBase CLI（`tcb`）或 SDK 批量上传 `choutuan-img/*` 到云存储桶，保留相对路径作为 objectKey（如 `choutuan-img/xxx.webp`）。
  2. 开启桶的 CDN / 自定义域名（生产必须，默认域名有带宽/防盗链限制）。
  3. 写一次性脚本：扫描 `assets` 表，将 `objectKey` 从本地路径更新为云存储 URL（`https://<cdn-domain>/choutuan-img/xxx.webp`）。
  4. 客户端图片组件改为读取 `asset.url` 而非本地 static 路径；并支持「本地兜底」便于未迁移时回退。
- **收益**：小程序主包体显著下降（760 webp 不再进包），加载走 CDN 更快。

### 4.4 一致性校验

- **行数比对**：各核心表 `count(*)` 本地 vs 云端一致（stores / menu_items / virtual_orders / accounts 等）。
- **金额一致性**：抽样虚拟订单，校验 `totalCents` 与订单行、配送费、打包费之和一致（沿用现有金额规则）。
- **资源可达性**：随机抽 50 个 `assets.url` 发起 HTTP HEAD，确认 200 且 Content-Type 正确。
- **空间字段**：GeoPoint 的 `coordSystem` 字段值无丢失（当前均为 gcj02）。
- **迁移回执**：生成 `migration-report.md` 记录导出时间、行数、文件数、校验结果。

## 5. 风险与回滚方案

### 5.1 风险点

| 风险 | 影响 | 应对 |
|---|---|---|
| 托管 PG 版本/扩展差异 | migration 执行失败 | P0 选定与本地一致的大版本；先在临时实例 dry-run migration |
| 云托管冷启动 / 实例拉起慢 | 首请求延迟 | 设最小实例数保活；压测首屏 |
| 微信登录 Secret 泄露 | 身份伪造 | 迁移到「微信云调用」，不在云托管环境变量存 Secret |
| 图片外链后小程序包体仍大 | 审核被拒 | P3 完成后重新构建并用开发者工具「代码依赖分析」确认主包 < 2MB |
| request 合法域名未配置 | 真机无法请求 API | P4 在小程序后台添加云托管域名（含 HTTPS） |
| 云存储默认域名限速/防盗链 | 图片加载慢或 403 | 生产绑定自定义域名 + CDN，配置跨域与 Referer 白名单 |
| 自定义域名需 ICP 备案 | 生产访问受阻 | 提前准备备案；过渡期用 `*.app.tcloudbase.com` 默认域名 |
| 成本超预期 | 预算风险 | 先用按量计费 + 监控告警；压测后定型套餐 |
| 数据导出导入丢行 | 功能异常 | 5.4 一致性校验 + 本地快照保底 |

### 5.2 回滚策略

- **基线保留**：本地 Docker 环境**全程不删除**，作为黄金副本。
- **API 回滚**：云托管支持多版本与流量切换。保留上一可用版本，异常时一键切回。
- **客户端回滚**：`VITE_API_BASE_URL` 改回 `http://127.0.0.1:3000`（或回退构建产物），重新构建发布。
- **数据回滚**：迁移前 `pg_dump` 全量快照存于本地；云端异常可 `pg_restore` 回本地继续服务。
- **开关设计**：客户端 API base 用构建变量 + 远端配置双保险，必要时远端下发旧地址。

## 6. 迁移后验证

### 6.1 功能验证（复用现有验收）

- `pnpm test` 单测通过；`pnpm typecheck` 通过；`pnpm build:mp-weixin` / `build:h5` 构建成功。
- 对照技术方案第 16 节 11 条 MVP 验收：
  - 游客不登录完成虚拟下单；
  - 登录后订单合并；
  - 地图真实、骑手路线虚拟且标注清晰；
  - 切后台 10s 恢复进度；
  - 购物车/结算/订单金额三处一致；
  - 无权利状态素材不返回；
  - 微信真机连续 3 单无阻断 Bug。
- 微信登录端到端（云调用）验证：游客 → 微信登录 → 订单合并。

### 6.2 性能测试

- **云托管**：记录实例冷启动耗时、API P95 延迟（对比本地 `127.0.0.1:3000` 基线）。
- **图片 CDN**：抽样图片首字节时间（TTFB）、加载成功率，对比本地 static。
- **并发**：用轻量压测（如 wrk/k6）打 `/v1/catalog/home`、`/v1/catalog/stores`，观察云托管自动扩缩与 PG 连接数。
- **小程序**：开发者工具 Audits 评分 > 90；主包 < 2MB；首屏 < 1.5s（中端安卓）。

### 6.3 数据与合规校验

- 5.4 一致性校验全部通过，出具 `migration-report.md`。
- 微信隐私协议、request 合法域名、云存储跨域配置复核。
- 微信登录改为云调用后，确认 `.env` 中 `WECHAT_MINI_APP_SECRET` 可从云托管环境变量移除。

## 7. 后续建议

1. **更新 ADR**：在 `docs/` 补一条架构决策记录，说明「以 CloudBase 作为托管平台，但业务仍为 NestJS + 关系数据库」，与第 3.2 节互为补充而非推翻。
2. **二期权密**：Redis/BullMQ、订阅消息、真实支付均在云托管 + 云调用框架下平滑扩展，无需返工。
3. **监控**：接入云开发监控/日志，配置 API 错误率与延迟告警。
4. **成本护栏**：设置每日费用告警，避免图片/CDN 流量失控。
