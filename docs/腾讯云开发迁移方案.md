# 白吃了 · CloudBase 原生架构决策

## 结论

项目全面切换到腾讯云开发原生架构：

- 小程序通过 `wx.cloud.callFunction` 调用 `api` 云函数。
- 后台通过 CloudBase 函数网关调用同一个 `api` 云函数。
- 运行时数据只存储在腾讯云开发文档数据库。
- 本地 NestJS、TypeORM、PostgreSQL、Docker Compose 链路不再参与开发、构建、验证或数据维护。

## 组件映射

| 组件 | 当前方案 |
|---|---|
| 小程序 | `apps/client`，默认 CloudBase API |
| 后台 | `apps/admin`，通过 CloudBase HTTP 网关访问 `/v1/admin` |
| 后端 | `apps/cloud-functions` 中的 `api` 云函数 |
| 数据库 | CloudBase 文档数据库 collections |
| 目录源数据 | `packages/catalog-data` |
| 类型契约 | `packages/api-contract` |

## 数据维护流程

店铺、分类、菜单改动后：

```bash
CLOUDBASE_ENV_ID=cloud1-d8g7o18ula3c12f10 pnpm cloudbase:seed
CLOUDBASE_ENV_ID=cloud1-d8g7o18ula3c12f10 pnpm cloudbase:verify
pnpm build:mp-weixin
```

如果云端 collections 还没有创建，先按 `apps/cloud-functions/src/collections.ts` 在 CloudBase 环境创建集合和索引。

## 旧链路处理

旧 PostgreSQL 方案已经废弃：

- 不再提供 `db:*` 脚本。
- 不再启动本地 API。
- 不再用 Docker Desktop 作为必要开发依赖。
- 不再把 `DATABASE_URL` 作为应用运行配置。

历史迁移 JSON 仍可通过 `cloudbase:import` 导入 CloudBase，但 PostgreSQL 不再是日常数据源或回滚目标。

## 验证边界

默认轻量验证：

```bash
pnpm --filter @baichile/cloud-functions typecheck
pnpm --filter @baichile/client typecheck
pnpm build:mp-weixin
```

深度测试、视觉化对比、真机全链路验收只在明确需要时执行。
