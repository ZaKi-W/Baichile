# PostgreSQL 持久化设计

## 目标

为现有 NestJS 后端接入 PostgreSQL/PostGIS，使账号身份、收货地址和虚拟订单在服务重启后仍可恢复，同时保持微信小程序现有 API 契约不变。

## 范围

本次持久化以下数据：

- 游客身份与微信账号映射
- 收货地址
- 虚拟订单及配送路线

商品目录继续使用 `catalog.seed.ts`。埋点仍只返回接收成功，不在本次落库。后台管理、支付、生产部署和云数据库采购不属于本次范围。

## 本地数据库

项目根目录新增 Docker Compose 配置，使用带 PostGIS 的 PostgreSQL 镜像。开发者通过一条命令启动数据库，数据保存在命名卷中。

本地连接参数写入 `.env.example`，实际密钥和连接串保留在已被 Git 忽略的 `.env`。应用统一读取 `DATABASE_URL`。

## TypeORM 集成

`AppModule` 通过 `TypeOrmModule.forRootAsync` 建立连接：

- 使用 `DATABASE_URL`
- 禁用 `synchronize`
- 使用显式实体列表
- 通过 migration 管理结构
- 非测试环境缺少连接串时启动失败并给出明确错误

迁移采用 TypeORM migration，不直接依赖手工执行 `infra/sql/001_init.sql`。已有 SQL 保留为早期架构参考，新的 migration 成为应用真实使用的数据库结构来源。

## 数据模型

### Identity

账号表保存稳定的内部账号 ID、微信 OpenID 摘要和展示资料。游客会话表保存 `visitorId` 及其可选的账号归属。

开发模拟登录同样创建账号记录。微信真实登录以 OpenID 摘要作为唯一身份，重复登录返回同一个账号，而不是每次创建新账号。

### Address

地址表保存当前 `Address` 契约中的字段，并通过 `visitorId` 或 `accountId` 标记归属。保存、列表、删除和游客转账号操作放在事务中执行；同一身份最多有一个默认地址。

### VirtualOrder

订单表保存报价快照、状态、身份归属、目的地及时间字段。订单明细和路线使用 JSONB 保存，避免当前 MVP 阶段过早拆分大量关联表。游客登录后，订单归属通过事务迁移到账号。

## 服务边界

- `AuthService` 负责创建和解析会话，并持久化身份。
- `AddressService` 只负责地址持久化和归属校验。
- `OrderService` 保留报价计算职责，并将创建、查询和合并切换到数据库。
- Controller API 路径和响应结构保持不变，只将相关方法改为异步。

Bearer Token 暂时保持现有格式，以控制本次范围。Token 签名、刷新和过期机制作为后续独立安全任务处理。

## 数据流

1. 小程序获取游客会话，后端写入游客会话记录。
2. 地址或订单通过 Bearer Token 解析身份并写入数据库。
3. 微信登录成功后创建或查找账号。
4. 同一事务内把该游客的地址、订单和游客会话迁移到账号。
5. 后续使用账号 Token 查询持久化数据。

## 错误处理

- 数据库不可用或配置缺失时，应用启动失败，不静默退回内存模式。
- 唯一约束和归属冲突转换为明确的 NestJS 业务异常。
- 查询不存在或不属于当前身份的数据时，不泄露其他用户数据。
- 数据库事务失败时整体回滚，避免只迁移地址或只迁移订单。

## 测试与验证

采用轻量验证，不执行深度测试：

- 为数据库服务行为增加集成测试，使用独立测试数据库。
- 覆盖创建游客、保存地址、创建订单、重启后读取和游客合并。
- 保留现有 API 测试，并修复 `apps/api` 测试脚本路径。
- 运行后端类型检查、构建和 API 测试。

测试不会使用开发数据库，也不会清空开发数据。

## 开发者操作

前端开发者只需：

1. 安装并启动 Docker Desktop。
2. 执行项目提供的数据库启动命令。
3. 复制示例环境变量。
4. 执行 migration 后启动 API。

具体命令会加入项目 README 或数据库说明文档。
