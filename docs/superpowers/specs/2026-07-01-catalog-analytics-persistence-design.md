# 商品目录与埋点持久化设计

## 目标

将目前仍由 TypeScript 种子和临时响应承载的服务端数据迁入 PostgreSQL，使分类、店铺、菜品和埋点事件全部由数据库持久化，同时保持微信小程序现有 API 契约不变。

## 范围

本次入库：

- 商品分类
- 店铺
- 菜品
- 菜品规格组与选项
- 埋点事件

本次不入库：

- 购物车草稿
- 当前定位与行政区缓存
- 当前选中的地址 ID
- 页面筛选、弹窗和加载状态
- 登录 Token 的客户端缓存
- 腾讯地图实时搜索结果

以上排除项属于客户端瞬时状态或外部服务结果，不应作为服务端业务主数据。

## 数据模型

### categories

保存分类 ID、名称、图标和排序值。分类 ID 沿用现有 API 中的稳定字符串。

### stores

保存现有 `StoreSummary` 的全部业务字段，包括分类归属、描述、标签、配送费用、起送金额、虚拟配送时长、销量、距离、评分、近期浏览人数、系统热度和来源类型。

`tags` 使用 PostgreSQL `text[]`。店铺与分类建立外键关系。

### menu_items

保存菜品 ID、店铺 ID、分类 ID、子分类 ID、名称、副标题、图片、基础价格、月销量和来源类型。

规格组及其选项使用 JSONB 保存。当前规格只随菜品整体读取和写入，没有独立检索或管理需求，拆成多张关联表会增加不必要复杂度。

### store_sub_categories

保存店铺内菜单子分类，包含店铺 ID、子分类 ID、名称和排序值。店铺与子分类使用联合唯一约束。

### analytics_events

保存事件 ID、游客 ID、账号 ID、事件名称、原始 payload 和创建时间。接口继续返回 `accepted` 与 `receivedAt`，但只有数据库写入成功才视为接收成功。

若请求体没有明确的 `eventName`，后端返回 400，而不是保存无法分析的匿名数据。

## 种子迁移

保留 `catalog.seed.ts` 作为开发环境的目录源和版本化初始数据。

新增幂等 seed 命令：

- 按稳定 ID upsert 分类、店铺、子分类和菜品。
- 重复执行不会产生重复记录。
- 已存在记录会更新为代码中的当前种子值。
- 不在 seed 中的数据库记录不会自动删除，避免误删未来后台创建的数据。

数据库 migration 只创建结构，不自动导入大批业务数据。首次运行流程为 migration 后执行 seed。

## 服务改造

`CatalogService` 注入分类、店铺、子分类和菜品仓储。`home`、分类列表、店铺列表、店铺详情和搜索全部改为异步数据库查询。

服务负责把多张表重新组装为现有 `StoreDetail`：

- 金额字段保持分为单位。
- 店铺标签恢复为字符串数组。
- 菜品规格 JSONB 恢复为 `SpecGroup[]`。
- 子分类按排序值返回。
- 搜索同时匹配店铺名和菜品名。

Controller 仅增加 `await`，路径、请求参数和响应字段不改变。

新增 `AnalyticsService` 负责校验并写入埋点，不让 Controller 直接操作仓储。

## 启动与开发流程

本地首次初始化：

```bash
pnpm db:up
pnpm db:migrate
pnpm db:seed
pnpm dev:api
```

`db:seed` 可以安全重复执行。README 更新为以上流程。

API 不在运行时静默回退到 TypeScript 目录；数据库为空时返回空目录，明确暴露漏跑 seed 的环境问题。

## 错误处理

- 数据库不可用时沿用当前启动失败策略。
- 目录查询不存在时继续返回 NestJS 404。
- 非法埋点请求返回 400。
- seed 在单个事务内执行，任何一条失败则整体回滚。
- 外键约束保证菜品、子分类和店铺不会引用不存在的分类或店铺。

## 测试与验证

采用轻量验证：

- Migration 测试确认新增表存在。
- Seed 测试确认重复执行后数量不翻倍。
- API 测试确认目录来自数据库，并在重启后保持一致。
- API 测试确认埋点写入数据库。
- 运行后端类型检查、构建和后端测试。

不运行完整前端测试、深度压力测试或视觉化对比。
