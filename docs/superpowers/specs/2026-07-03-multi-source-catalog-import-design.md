# 多来源中文菜单导入设计

## 目标

将 Uber Eats、Talabat 和 ChowNow 三批抓取数据加入现有目录，同时剔除缺少图片的商家和菜品、避免重复，并在写入数据库前把商家展示名和菜品名称中文化。

## 输入

- `/Users/wangzijin/Documents/MyCode/Douyin-copy/output_ubereats`
- `/Users/wangzijin/Documents/MyCode/Douyin-copy/output_talabat`
- `/Users/wangzijin/Documents/MyCode/Douyin-copy/output_chownow`

三个目录使用相同的 `metadata.json` 结构。导入器读取各商家子目录，不依赖聚合 library 文件。

## 过滤与去重

1. 商家必须有非空 `cover`，并且过滤后至少有一个有效菜品，否则整店跳过。
2. 有效菜品必须同时满足：
   - `has_local_image === true`
   - `image_url` 非空
   - `local_image_path` 非空且文件存在
3. 新数据内部及数据库现有商家按标准化原始商家名去重。标准化时忽略大小写、标点和多余空白。
4. 同一商家内的菜品按“标准化原始菜名 + 分价价格”去重。同名不同价保留；完全重复时优先保留描述和图片信息更完整的记录。
5. 去重在翻译前完成，避免不同原文偶然得到相同中文译名后被误删。
6. 来源和原始 ID 共同组成稳定导入 ID，使重复执行保持幂等，并避免不同平台 ID 碰撞。

## 中文化

- 商家优先使用通行官方中文名；没有通行中文名时保留品牌原文，只翻译地点、门店等后缀。
- 菜品名称完整翻译为简体中文；品牌、专有名词和规格信息尽量保留。
- 翻译结果生成并保存为离线缓存，以“来源 + 原始 ID + 原文”定位。数据库导入只读取缓存，不在事务中调用网络服务。
- 已缓存译文可重复使用；原文变化时视为新条目并重新生成，避免旧译文误配。
- 无法可靠翻译的名称保留原文，不阻断整批导入。

## 导入流程

1. 扫描三个输入目录并解析元数据。
2. 校验图片并过滤无图记录。
3. 以原文执行商家与菜品去重。
4. 读取或生成中文翻译缓存。
5. 构建统一目录记录并与数据库现有商家去重。
6. 在单个数据库事务中写入分类、商家、店内分类和菜品。
7. 输出每个来源的读取数、过滤数、去重数、翻译数和最终写入数，便于核对。

现有 `derived` 导入数据由本次三来源全集原子刷新；非 `derived` 数据不删除。遇到与非导入商家同名的记录时跳过新商家，避免覆盖人工维护内容。

## 验证

只做轻量验证：

- 单元测试覆盖无图过滤、商家去重、菜品“名称 + 价格”去重和翻译缓存应用。
- 运行导入器的预检模式，核对三来源统计。
- 执行真实导入后查询数据库数量，并抽查中文商家名、中文菜品名以及图片字段。
- 不进行深度测试或视觉化对比。
