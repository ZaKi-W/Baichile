# 白吃了后台管理端设计

## 1. 目标与范围

为“白吃了”微信小程序提供一个仅供内部人员使用的 Web 后台。第一阶段交付可实际运行的管理闭环，而不是静态界面：

- 管理员登录、退出、会话保持与修改自己的密码。
- 首页仪表盘展示商家、菜品、用户、订单和货币概览。
- 商家、菜品、用户信息、用户货币、用户订单的查询与管理。
- 超级管理员、运营、客服三级角色权限。
- 所有敏感写操作写入审计日志。

后台不向商家开放，不实现商家租户隔离、商家注册、审批工作流、批量导入、图片上传、复杂报表和自定义角色编辑器。这些能力留给后续阶段。

## 2. 方案比较与选择

### 方案 A：Monorepo 内新增独立 Vue 管理端，共用现有 API（采用）

在 `apps/admin` 新建 Vue 3 + Vite + TypeScript 单页应用；在现有 NestJS 服务中新增 `/v1/admin` API、后台鉴权、RBAC 和审计模块；继续使用同一个 PostgreSQL 数据库。

优点是复用当前 Vue/TypeScript 技术栈、实体与部署方式，前后端边界清晰，改动集中且后续可独立部署。缺点是需要新增一个前端应用并维护后台专用 API。

### 方案 B：在小程序客户端中增加后台页面

开发量看似较少，但会把内部管理能力打包进面向用户的小程序，权限边界、发布节奏和桌面端操作体验都不合适，因此不采用。

### 方案 C：使用低代码后台

可以更快生成基础 CRUD，但权限、钱包账务、订单业务规则和审计需要大量定制，也会引入新的运行平台，因此当前阶段不采用。

## 3. 总体架构

后台采用模块化单体架构：

```text
apps/admin（Vue SPA）
        │ Bearer Token
        ▼
apps/api /v1/admin（NestJS）
        │
        ├─ AdminAuth：登录、会话、密码
        ├─ AdminRbac：角色与权限校验
        ├─ AdminCatalog：商家与菜品
        ├─ AdminAccounts：用户资料
        ├─ AdminWallet：余额调整与流水
        ├─ AdminOrders：订单查询与状态操作
        ├─ AdminDashboard：汇总指标
        └─ AdminAudit：敏感操作审计
        │
        ▼
PostgreSQL / TypeORM
```

后台与小程序共享业务数据，但后台接口独立于现有用户接口。管理写操作由服务层统一校验，前端是否隐藏按钮不作为权限保障。

## 4. 技术选型

- 前端：Vue 3、Vite、TypeScript、Vue Router、Pinia。
- UI：Element Plus。后台表格、表单、分页、弹窗和反馈组件优先使用框架原生能力。
- 请求：原生 `fetch` 封装统一的 token、错误和查询参数处理。
- 后端：现有 NestJS + TypeORM。
- 鉴权：管理员账号密码登录；密码使用 `scrypt` 加盐哈希；服务端保存哈希后的不透明会话 token。
- 数据库：继续使用现有 PostgreSQL，通过 TypeORM migration 增加后台表和业务字段。

## 5. 权限模型

采用固定三级角色，不在第一阶段提供角色配置页面：

| 能力 | 超级管理员 | 运营 | 客服 |
|---|---:|---:|---:|
| 查看仪表盘 | 是 | 是 | 是 |
| 查看商家/菜品 | 是 | 是 | 是 |
| 新增、编辑、上下架商家/菜品 | 是 | 是 | 否 |
| 查看用户基础资料 | 是 | 是 | 是 |
| 编辑用户昵称、状态 | 是 | 否 | 是 |
| 查看钱包余额/流水 | 是 | 是 | 是 |
| 调整用户余额 | 是 | 否 | 否 |
| 查看订单 | 是 | 是 | 是 |
| 修改订单管理状态/备注 | 是 | 是 | 是 |
| 管理管理员账号 | 是 | 否 | 否 |
| 查看审计日志 | 是 | 否 | 否 |

权限在 API 路由上以明确 permission key 校验。禁用的管理员或过期、撤销的会话不能访问任何后台 API。

## 6. 数据模型调整

### 6.1 新增后台表

`admin_users`

- `id`
- `username`（唯一）
- `display_name`
- `password_hash`
- `role`：`super_admin | operator | support`
- `status`：`active | disabled`
- `last_login_at`
- `created_at`
- `updated_at`

`admin_sessions`

- `id`
- `admin_user_id`
- `token_hash`（唯一）
- `expires_at`
- `revoked_at`
- `created_at`

`admin_audit_logs`

- `id`
- `admin_user_id`
- `action`
- `resource_type`
- `resource_id`
- `before_data`（JSONB）
- `after_data`（JSONB）
- `ip_address`
- `created_at`

### 6.2 现有业务表补充字段

- `stores`：增加 `status`、`created_at`、`updated_at`。
- `menu_items`：增加 `status`、`created_at`、`updated_at`。
- `accounts`：增加 `status`、`updated_at`。
- `virtual_orders`：增加 `admin_status`、`admin_note`、`updated_at`。

状态字段使用受控字符串值，避免后台真正删除仍被订单引用的数据。商家和菜品采用上下架；用户采用正常/禁用；订单保留原有虚拟配送状态，并用独立管理状态和备注记录人工处理，不伪造配送过程。

钱包余额继续以 `accounts.balance_cents` 为当前余额，以 `wallet_transactions` 为不可变流水。后台调整余额必须在数据库事务中同时更新余额和新增 `admin_adjustment` 流水，禁止直接覆盖余额。

## 7. 功能设计

### 7.1 登录与后台外壳

- 用户名、密码登录。
- 登录成功后加载当前管理员与权限。
- 左侧菜单按权限显示，顶部展示当前账号与退出入口。
- API 返回 401 时清理会话并跳转登录页；403 显示无权限。
- 超级管理员可以新增、编辑、禁用管理员和重置密码。

系统通过环境变量创建第一个超级管理员，避免在仓库中保存默认密码。

### 7.2 仪表盘

展示：

- 商家总数、上架商家数。
- 菜品总数、上架菜品数。
- 用户总数、今日新增用户。
- 订单总数、今日订单数、按管理状态汇总。
- 用户货币总余额、今日货币增减额。

第一阶段使用数据库聚合实时查询，不引入缓存和离线报表。

### 7.3 商家管理

- 按名称、分类、状态查询并分页。
- 查看商家详情。
- 新增、编辑商家。
- 上架、下架商家。
- 商家存在订单时不物理删除。

编辑字段覆盖当前 `StoreEntity` 的可运营属性。数值在界面使用元或常规单位展示，在 API 边界转换为数据库单位。

### 7.4 菜品管理

- 按名称、所属商家、分类、状态查询并分页。
- 查看、新增、编辑、上下架菜品。
- 编辑基础价格、热量、图片 URL、销量、排序和规格组。
- 规格组先使用结构化 JSON 编辑区并进行 schema 校验，避免第一阶段实现复杂规格设计器。

菜品必须引用存在的商家；商家下架不自动修改菜品状态，但小程序目录查询只展示商家和菜品均为上架的数据。

### 7.5 用户管理

- 按 ID、昵称、状态查询并分页。
- 查看昵称、头像、注册时间、账号状态、余额、订单数。
- 编辑昵称和账号状态。
- 禁用用户后，其既有数据保留；用户鉴权与下单接口拒绝被禁用账号。

微信 OpenID 哈希只用于精确内部定位，不在列表中完整展示，也不允许后台修改。

### 7.6 用户货币管理

- 查看用户当前余额和钱包流水。
- 按流水类型、时间范围查询并分页。
- 超级管理员可以执行增加或扣减，必须填写原因。
- 金额必须为正整数分；扣减后余额不能小于零。
- 每次调整产生不可变 `admin_adjustment` 流水并写审计日志。
- 不支持修改或删除历史流水。

### 7.7 用户订单管理

- 按订单号、用户、商家、配送状态、管理状态、创建时间查询并分页。
- 查看订单商品、金额、热量、地址标识、路线摘要、异常与退款信息。
- 客服及以上可修改独立的管理状态：`normal | following_up | resolved`。
- 客服及以上可编辑内部备注。
- 不允许后台修改订单金额、商品快照、创建时间或虚拟配送时间线。

### 7.8 审计日志

记录登录、管理员变更、商家/菜品写操作、用户状态变更、余额调整、订单管理状态和备注变更。审计日志仅超级管理员可查看，不允许从后台删除。

## 8. API 约定

后台接口统一位于 `/v1/admin`：

- `/auth/login`、`/auth/logout`、`/auth/me`、`/auth/change-password`
- `/dashboard`
- `/admin-users`
- `/stores`
- `/menu-items`
- `/accounts`
- `/accounts/:id/wallet`
- `/accounts/:id/wallet/adjustments`
- `/orders`
- `/audit-logs`

列表接口统一接收 `page`、`pageSize`、`keyword` 和模块筛选参数，统一返回：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0
}
```

错误统一返回稳定的 `code` 和中文 `message`。所有输入使用 DTO 校验；未知字段被剔除或拒绝。写操作返回更新后的资源。

## 9. 数据流与一致性

1. 管理员登录后，API 创建会话并只向前端返回一次明文 token。
2. 前端将 token 保存在会话存储中，请求时放入 `Authorization: Bearer`。
3. API 对 token 哈希后查询会话，校验有效期、撤销状态、管理员状态和权限。
4. 普通 CRUD 在服务层校验并写库。
5. 钱包调整使用单个数据库事务锁定用户记录、校验余额、写余额和流水，再写审计。
6. 写操作成功后前端重新获取当前页，避免本地状态与数据库漂移。

## 10. 错误处理与安全

- 登录失败统一提示账号或密码错误，不泄露账号是否存在。
- 登录接口按 IP 和用户名进行进程内基础限流；生产部署时再升级为 Redis 限流。
- 密码至少 10 位；会话默认 8 小时，修改密码后撤销该管理员的其他会话。
- 金额转换使用整数分，禁止浮点金额落库。
- 404 表示资源不存在，409 表示唯一性或业务状态冲突，422 表示输入不合法。
- 前端对删除、禁用、下架、余额调整等操作二次确认。
- 审计记录不保存密码、token、完整 OpenID 哈希等敏感信息。
- 后台生产环境要求 HTTPS，并通过环境变量限制允许的管理端来源。

## 11. 验证策略

遵循项目要求，只做与改动直接相关的轻量验证：

- API：后台鉴权、RBAC、钱包事务、关键筛选与状态变更的定向测试。
- 管理端：类型检查、构建，以及请求封装和权限判断的少量单元测试。
- 数据库：migration up 验证与实体元数据测试。
- 小程序：构建微信小程序目标，确认目录过滤和用户禁用逻辑未破坏主链路。

不执行全量深度测试，不做视觉化对比。

## 12. 实施分段

1. 后台数据模型、migration、管理员鉴权、RBAC 和审计基础设施。
2. 后台管理 API：仪表盘、商家、菜品、用户、钱包、订单、管理员和审计。
3. `apps/admin` 工程、登录、布局、路由、请求和权限框架。
4. 各业务管理页面与表单。
5. 小程序目录过滤、禁用用户保护、轻量回归和版本号自增。

每个阶段保持现有小程序优先：任何共享业务修改都必须确认微信小程序构建可用。整个代码实现任务只将 `apps/client/src/config/code-version.ts` 的 `CODE_VERSION` 自增一次。
